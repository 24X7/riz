// where riz is going

From Lambda runtime to agent substrate.

riz started as the Lambda contract you already know — exact HTTP API v2 + WebSocket payloads, real Lambda context, your handlers unchanged. But the moment those functions are running, they're also tools: an MCP server agents speak natively, an OpenAI-compatible gateway routing every model call, and a WASI sandbox built to run code you didn't write. The runtime and the agent substrate are the same binary — that's the point.

live · WASM

Capability-sandboxed runtime

Run a runtime = "wasm" handler under wasmtime with deny-by-default filesystem and network — capabilities granted explicitly in riz.toml. Sub-ms cold starts on code you didn't write, parity-tested alongside Bun, Node.js, Python, and Rust.

live · gateway

One LLM gateway, every model

Handlers call ctx.invokeModel(name, prompt) — riz routes across OpenAI, Anthropic, and local Ollama by model prefix, with budget caps, fallback, SSE streaming, embeddings, and per-provider cost telemetry. An OpenAI-compatible endpoint at /_riz/v1/* means existing clients just work.

v0.2 · replay

Record & replay for agents

Every request captured with input, output, timings, and downstream calls. riz replay --since 1h --function chat replays N events against a hot-reloaded handler and diffs the responses. The agent can ship a change, run the replay, and decide if anything regressed — without writing tests.

One file.
Five runtimes.

Declare your entire function fleet in a single riz.toml. One [function.<name>] block is one process pool serving N routes — mirroring AWS exactly. Routing, process management, hot-reload, and resource limits are automatic.

Same wire format AWS sends, parity-tested across Bun, Node.js, Python, Rust, and a wasm32-wasip1 module under wasmtime. Point any HTTP client at riz the way you would at API Gateway — same payloads, different bill.

runtimes bun node python rust wasm
protocols http api v2 websocket
riz.toml toml 
# one function = one process pool = N routes
# mirrors AWS Lambda + API Gateway v2

[function.api]
runtime     = "bun"
handler     = "src/api/index.handler"  # AWS file.export
timeout_ms  = 30000
concurrency = 10

# Implicit default: ANY /api
# Or explicit, with AWS-style path params + greedy proxy:
[[function.api.routes]]
path   = "/api/{id}"
method = "GET"

[[function.api.routes]]
path   = "/api/{proxy+}"
method = "ANY"

Features

v0.1 · in production

Agent & AI Integration

  • MCP server at /_riz/mcp — every function is a tool, automatically, spec 2025-11-25
  • Zero SDK code — point any MCP client at riz, your APIs become agent-callable
  • Bearer-token gating on /_riz/mcp for shared-secret access control

LLM Gateway

  • OpenAI-compatible gateway at /_riz/v1/* — provider routing, fallback, SSE streaming, embeddings
  • Route across OpenAI, Anthropic, and local Ollama by model prefix
  • Budget caps + per-provider cost telemetry — AI-FinOps where you already watch latency

Function Runtimes

  • Five runtimes — Bun, Node.js, Python, Rust, and capability-sandboxed WASM
  • Cross-runtime parity matrix — every HTTP capability tested on every runtime
  • WASI sandbox (runtime = "wasm") — deny-by-default fs/net, sub-ms cold starts

Protocol Surface

  • AWS API Gateway v2 HTTP payload — exact aws_lambda_events types
  • WebSocket APIs ($connect / $disconnect / $default) + @connections management API at /_riz/connections/{id} — handlers in Bun, Python, and Rust
  • WebSocket connection list endpoint — GET /_riz/connections

Configuration & Routing

  • Function-centric config: [function.<name>] + N routes per function
  • AWS path syntax: {id}, {proxy+}, $default
  • AWS handler syntax: handler = "index.handler"

Deployment & Lifecycle

  • Hot-swap deploys from S3 with in-flight request drain
  • riz.toml hot-reload on save
  • Real Lambda context — getRemainingTimeInMillis, functionName, invokedFunctionArn, awsRequestId

Observability

  • /_riz/health · /_riz/metrics · /_riz/registry
  • Terminal dashboard: P50/P75/P90/P95/P99 over 5-min window
  • Datadog metrics emitter
  • MCP server — spec 2025-11-25 (also negotiates 2024-11-05 / 2025-03-26 / 2025-06-18). JSON-RPC 2.0, full lifecycle.

Developer CLI

  • riz init <template> — scaffold a working project (7 templates: TS/Python/Rust/Node × HTTP, TS/Python/Rust × WebSocket)
  • riz run — boot the runtime with riz.toml + handler-source hot reload
  • riz validate / riz routes — config check + route table
  • riz deploy — S3 hot-swap deploy with auto health-check rollback

Security & Auth

  • Lambda authorizers — REQUEST + JWT (JWKS, TTL cache)
  • CORS auto-preflight — OPTIONS → 204, Access-Control-Allow-* headers
  • Bearer-token auth on /_riz/* (/_riz/health stays open)

Security & Isolation

  • Always-on safety profile per child — RLIMIT_CORE=0, FD caps, fork-bomb caps, no privilege escalation
  • Opt-in per-function caps — memory_mb, cpu_time_secs, and Linux Landlock filesystem allowlists
  • WASI sandbox — deny-by-default filesystem and network, capabilities granted explicitly in riz.toml
  • Lambda authorizers (REQUEST + JWT/JWKS), CORS auto-preflight, bearer-token admin gating

Proof

  • Production-readiness bug tracker closed, with regression-gate tests for the fixed bugs

How riz compares

If you're weighing riz against the usual local-Lambda options, here's the short version. riz is the only one that's a single Rust binary, the only one with a current-spec MCP server built into the runtime, and the only one with OS-level handler sandboxing — while running your existing Lambda code unchanged.

Capability riz LocalStack SAM Local Cloudflare Workers
Surface HTTP API v2 + WebSocket Lambda Full AWS emulation Lambda + API Gateway Workers runtime (different model)
Per-request cost None — process pool Docker container Docker container Edge compute
Cold start ~50 ms (process spawn) Seconds (Docker) Seconds (Docker) ~5 ms (V8 isolate)
Built-in MCP server Spec 2025-11-25 No No No
Existing Lambda code unchanged Yes (exact API GW v2 payload) Yes Yes No (different model)
Single binary Yes (~10 MB Rust) No No No (managed service)
OS-level handler sandbox Yes (Landlock + rlimits) No No Via V8 isolate
Self-host in prod Yes — primary use case Possible (overkill) No No (it IS the cloud)

Short version: LocalStack is all of AWS on your laptop, slowly; SAM Local is AWS's emulator for CloudFormation users; Workers is the cloud. riz is just the Lambda runtime — on your box, fast, MCP-native. Questions about a specific head-to-head? chris@riz.dev.
