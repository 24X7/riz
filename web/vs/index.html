Riz vs LocalStack

// when each one wins

Pick Riz when

  • You only need HTTP API v2 and WebSocket Lambdas. The rest of AWS is not in your dev loop.
  • You want sub-second cold starts. LocalStack spins Docker containers per invocation; Riz keeps process pools warm.
  • You want a live terminal dashboard with real percentile latency, not log greps.
  • You want every function to be an MCP tool the moment riz run boots.
  • You want to run this in production, not just local dev.

Pick LocalStack when

  • You need full AWS service emulation: SQS, SNS, S3, DynamoDB, IAM, CloudFormation, the works.
  • You're building a multi-service application and need to test cross-service IAM policies, S3 event bindings, etc.
  • Your CI already runs localstack start and you don't want to re-tool.
  • You're fine with Docker overhead and the per-invocation cold-start cost.

// the short version

LocalStack is "all of AWS, on your laptop, slowly." Riz is "just the Lambda runtime, on your server, fast."

Most teams need both: LocalStack for the rare cross-service integration test, Riz for daily Lambda dev and self-hosted production.

Riz vs SAM Local

// when each one wins

Pick Riz when

  • You don't already live in CloudFormation. SAM Local assumes a SAM template; Riz reads a 10-line riz.toml.
  • You want hot-reload that swaps the process pool in under a second. SAM Local spawns a fresh Docker container per call.
  • You want to actually run the runtime in prod. SAM Local is a dev-only tool.
  • You want the MCP integration story without writing it yourself.
  • You're tired of waiting for Docker images to build.

Pick SAM Local when

  • You're already deep in CloudFormation/SAM templates and want AWS-tooling-compatible local invocation.
  • You need to exercise a Lambda layer or runtime-specific Lambda Extension (Riz doesn't ship those).
  • You're invoking your Lambda from CodeBuild/CodePipeline locally with the same packaging.

// the short version

SAM Local is "AWS's local-emulator for AWS's CloudFormation users." Riz is "a real runtime that happens to speak Lambda's wire format."

If you'd rather treat Lambdas as code than as CloudFormation resources, Riz is the better dev loop.

Riz vs Cloudflare Workers

// when each one wins

Pick Riz when

  • You have AWS Lambda code today and want to keep it. Workers requires re-writing to a different runtime.
  • You need to stay on your own infrastructure — compliance, data residency, egress cost, vendor independence.
  • You want OS-level isolation per handler (Landlock + rlimits), not just V8 isolates.
  • You need WebSocket lifecycle ($connect/$disconnect/$default) with the AWS API GW v2 connections protocol.
  • You want every function to be an MCP tool natively, not by writing a Workers binding.

Pick Workers when

  • You want edge compute — code running close to every user, automatically.
  • You're starting greenfield and don't mind writing to Workers's API instead of Lambda's.
  • You want sub-10 ms cold starts and your handler fits the V8 isolate model.
  • You don't want to operate any infrastructure at all.

// the short version

Workers is the cloud. Riz is your box. The trade is operational ownership for control, predictability, and zero vendor lock.

If your AWS bill is the reason you're reading this page, Riz is the answer. If you want zero ops, Workers is.
