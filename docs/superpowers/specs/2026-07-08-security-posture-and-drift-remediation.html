Package B

BSupply-chain guardrails

Today: no cargo-deny, no cargo audit, no SBOM, no signing. The security page can only claim what CI actually enforces, so the machinery lands before the page.

  • B1 — deny.toml + CI gate. Four checks (advisories · licenses · bans · sources). Gated once the tree is clean.
  • Discovery risk. cargo-deny may surface a real advisory. Handling, in order: (1) update the dep; (2) if unfixable, a dated, documented deny.toml ignore with the RUSTSEC id. CI stays green; the state is recorded.
  • B2 — CycloneDX SBOM generated on release, attached as riz.cdx.json. No gating.
  • B3 — build-provenance via actions/attest-build-provenance (keyless; id-token: write + attestations: write). No secrets to hold.
Deferred to roadmap — documented on the C page, not built here

Detached cosign signatures · the curl … | sh installer verifying before it executes (touches the outward-facing hosted install path — larger, riskier, its own effort) · external security audit.

Package C · the design target

CThe security-posture page

A new web/security.html in the established site style — the perimeter counterpart to sandbox.html. Where sandbox sells the capability, this page states where the confinement stops. Every protection statement links a real test or code path.

C · §2  The isolation matrix

The centerpiece. The one truth the marketing page glosses: strong deny-by-default isolation is a WASM property, not a whole-runtime one. Native runtimes get resource ceilings and (on Linux) a filesystem allowlist — not a syscall or network jail.

C · §3  Three walls — with their limits

Cross-links sandbox.html, then states each wall's edge plainly instead of leaving it implied.

C · §4  Threat model & trust boundaries

Assets, the boundaries that protect them, and what sits in scope. Reuses SECURITY.md's scope section; notes the fail-closed /deploy gate (503 with neither key nor CIDR).

C · §5–6  Supply chain & reporting

  • Supply chain — cites Package B as shipped: cargo-deny gate, SBOM per release, build-provenance + the gh attestation verify invocation. Roadmap subsection names cosign, verify-on-install, external audit.
  • Reporting — links SECURITY.md (private advisory / chris@riz.dev, 3-day ack).
  • Wiring — nav + footer on every page, cross-link from sandbox.html, add to llms.txt / .well-known/riz.json, register concrete claims in tests/claims/registry.toml.

Every protection claim maps to a referent — e.g. child_inherits_always_on_caps · child_with_allowlist_can_read_allowed_path_only · src/deploy.rs fail-closed gate. No claim without one. No banned framing in copy.
