Lambda‑shape purity & the WASM capability suite

Every function is authored as a Lambda handler; WASM guests reach Postgres, HTTP, DynamoDB, Supabase and Clerk through a daemon‑brokered capability layer that never lets a credential cross the sandbox. Synthesized from three independent designs (minimalist / capability‑platform / DX‑first) scored by a three‑judge adversarial panel, with every load‑bearing repo claim verified against source.

Ground truth

Requirements — source of record

R1 · purity
Every function is authored as a Lambda-style handler: APIGW v2 event in, Lambda response out. The stdin line-JSON programming model dies as an authoring contract — authors never write an event loop. Shims/SDKs own the loop. All examples conform; a conformance test enforces it.
R2 · capabilities
A brokered host capability layer for wasm32-wasi guests: outbound HTTP, Postgres via a host-side sqlx pool, DynamoDB (SigV4), Supabase, Clerk. Secrets are brokered by the host — guests never hold raw credentials. Capabilities are declared per-function in riz.toml.
R3 · separation
templates/ (consumed by scaffolding) covers exactly six targets: wasm (rust), rust, go, typescript (bun), typescript (node), python. examples/lambdas/* become showcase apps that conform to R1 — proof, not starting points.
R4 · scale
The gateway→runtime and guest→host call paths (wire format, pooling, concurrency, backpressure) are designed for running riz exclusively at scale — especially WASM capability calls.

Constraints carried through the whole design: the AWS Lambda + API GW v2 wire contract is sacrosanct at the HTTP boundary; docs/SAFETY.md binds all of src/; reject-not-queue backpressure everywhere; cargo-nextest only; site claims land only with pinned tests (tests/claims/registry.toml).

Verified against source

Current state

The audit's central surprise: the architecture already enforces R1 everywhere except WASM — and upstream #60 proved the gap by shipping a template with yet another hand-written stdin loop.

  • The broker beachhead is real and end-to-end tested — but per-child. The single-seam dispatcher (deny-by-default, payload caps, token bucket, reject-not-queue inflight, deadline, audit) lives in src/broker/mod.rs, backed by one unpooled tokio-postgres client per wasm worker (src/broker/pg.rs:27-33). Guests reach it through raw riz_broker extern imports with the two-call pg_query/read_response stash pattern. Limits silently multiply by concurrency; credentials ride every child's env (RIZ_BROKER_RESOURCES) with full-env inheritance; there is no shared pool.
  • The config seam already exists: [resources.pg.*] + [function.X.capabilities.*] (CapabilityGrant at src/config.rs:354, CAPABILITY_TYPES at :402).
  • WASM execution: wasm32-wasip1 sync core modules under wasmtime 45, one instance per pool worker for the worker's life, full Cranelift JIT at every spawn (Engine::default() + Module::from_file, wasm.rs:219-220), run in a riz __wasm-host child so the capability boundary is an OS process boundary.
  • Transport today: stdio line-JSON one-line-in/one-line-out, strictly one in-flight per worker, semaphore try_acquire admission with an O(n) idle-worker scan that can mis-serve two permit-holders (src/process/mod.rs:195-214 — a verified correctness bug, not just a perf smell). Rust/Go ride the Runtime API. Every overload path already answers with explicit backpressure.
  • Templates: 9 exist. The newest, wasm-http (#60, landed after this design's audit began), is authored as the hand-written stdin loop R1 abolishes; typescript-http is Bun in disguise; nodejs-http is plain .mjs JavaScript; three websocket variants sit outside the six targets; riz init BUILTINS already leak two examples into the scaffold list (src/template_fetch.rs).
R1 — Lambda-shape purity

riz-wasm and the authoring contract

crates/riz-wasm — a workspace member, published to crates.io before any template depends on it. The API deliberately mirrors lambda_runtime so Rust authors transfer directly from the rust template. A user's whole file is fn handler(...) plus fn main() { riz_wasm::run(handler) }.

// Value-first API — guaranteed path, no feature flags:
pub fn run(handler: fn(Event, Context) -> Result<Response, Error>) -> !
// Event/Response are serde_json::Value newtypes with accessors; Context carries the
// function name, request id (read from event.requestContext.requestId, empty string
// when the event lacks it), and the deadline (remaining_time()).

// Typed events behind a default-on `typed-events` feature, ONLY if aws_lambda_events
// compiles cleanly to wasm32-wasip1 (chrono/http feature graph must be CI-verified
// before the feature ships; the Value path is the guaranteed fallback either way):
pub fn run_typed(
    handler: fn(LambdaEvent<ApiGatewayV2httpRequest>) -> Result<ApiGatewayV2httpResponse, Error>,
) -> !

Mechanics the shim owns (and deletes from every guest)

  • The event loop. v1 speaks the current line-JSON envelope ({event, __riz_deadline_ms, __riz_function_name}) so PR1 touches zero src/ transport code.
  • Wire negotiation, driven by the guest's marker. At module load the host reads the shim's exported ABI marker (riz_abi_v1 now, riz_abi_v2 once the framed wire ships), selects the highest wire the guest supports, and sets RIZ_WIRE to the selected version at spawn; the shim treats RIZ_WIRE as authoritative (absence = line-JSON v1); a missing or unknown marker fails closed at load with an actionable error instead of a hung pipe. Compiled .wasm artifacts outlive riz upgrades on the invocation wire — negotiation is mandatory, not optional. (The capability ABI makes one sanctioned pre-1.0 break in PR4.)
  • The capability ABI. Typed riz_wasm::cap::{pg, http, dynamo, supabase, clerk} clients wrap the raw imports; guests never write unsafe extern blocks.
  • The marker itself. riz_abi_v1 is an exported symbol (survives wasm-opt better than custom sections), doubling as an artifact-level conformance signal.

Rust and Go need no shim: the official AWS SDKs against riz's Runtime API are the R1 contract for static binaries, and replacing them would trade away the run-unmodified-on-AWS claim for nothing.

R2 — WASM capability suite

The broker moves into the daemon

This topology change is what makes three sentences of R2 structurally true rather than aspirational: a shared host-side pool exists, max_inflight/ rate_per_sec finally mean per-function (today they silently multiply by concurrency), and credentials exist in exactly one process.

Auth — never caller-supplied identity

  • At spawn the daemon issues each granted worker a random 32-byte token (RIZ_BROKER_TOKEN + RIZ_BROKER_SOCK); the daemon maps token→function. SO_PEERCRED uid check layered on top; socket dir 0700, socket 0600.
  • Every worker runs as the same uid, so a function-name handshake would let any compromised worker — including native bun/node ones — claim another function's grants. Tokens, not names.
  • Grantless functions get no socket and no token; the host import answers the structured denied envelope locally with zero IPC.

BCP — broker call protocol

The dispatcher seam — unchanged ordering, now per-function

Guest ABI v2 — one dispatcher import

The per-verb imports collapse to riz_capability.call(verb_ptr, verb_len, grant_ptr, grant_len, req_ptr, req_len) → i32 plus the existing read_response stash pattern, with the same guest_range bounds checks. Collapsing happens only after PR1 has migrated every non-fixture guest onto riz-wasm; the raw-extern fixtures (broker-wasm, guard-wasm) are kept raw deliberately — they are the adversarial coverage of the guest_range/fail-closed ABI paths — and update alongside the import they exist to attack.

Secrets brokering

  • All credentials resolve inside the daemon at startup (fail-fast on missing env vars for any referenced resource) and never enter child processes, argv, config values, or guest memory. RIZ_BROKER_RESOURCES and --broker-grants are deleted.
  • Children get a scrubbed env: env_clear() then an explicit per-runtime allowlist — PATH, HOME (bun/node resolution), SSL_CERT_FILE, locale/proxy vars where needed, AWS_LAMBDA_* for RuntimeApi workers (env_clear must precede the explicit .env() calls or Rust/Go pools break at spawn), RIZ_* control vars, plus the function's declared env/stage_variables as the documented escape hatch.
  • Proven by an all-six-runtimes e2e smoke in the same PR and a secrets canary test: a canary DSN asserted absent from guest-observable bytes and from a dump of the child's actual process environment.

The five capability classes

Verb namespace <type>.<op>. Grant vocabulary — mode, max_inflight, rate_per_sec, call_timeout_ms, max_request_bytes, max_response_bytes — reused verbatim from CapabilityGrant; CAPABILITY_TYPES widens to ["pg","http","dynamo","supabase","clerk"].

Per-type scoping keys extend the grant, never the resource: http grants add methods; dynamo grants add key_prefix (optional partition-key prefix, enforced before signing); supabase grants add path_prefix + methods; clerk grants add operations. Resources hold connection and credential config only. Grant and resource structs gain #[serde(deny_unknown_fields)] so a misplaced key fails at config load instead of parsing as silence.

http hardening — mandatory for any http grant

The default grant shape is resource-pinned: the guest supplies a relative path under the resource's base_url and never holds an absolute URL; allow_origins (exact scheme+host+port) exists as the generic mode. Either way, the daemon resolves DNS and refuses private/link-local IPs after resolution (DNS-rebinding an allowlisted host into 10.0.0.1 is otherwise an internal-network probe), redirects are re-validated against the same rules, and methods allowlists apply. Response bodies respect max_response_bytes with no streaming in v1.

R4 — call transmission at scale

RWP — the framed invocation wire

For stdio runtimes (bun, node, python, wasm, guards). Rust/Go keep the per-worker AWS Lambda Runtime API unchanged — the "unmodified official binaries" claim is load-bearing, and Lambda execution semantics forbid pipelining there; they scale via pool size.

Perf levers, in order of leverage

1 · serialize once
The gateway serializes the APIGW event to bytes once; those exact bytes are the INVOKE payload. Deletes one full serde re-encode each way (today's build_envelope_payload struct→envelope pass).
2 · admission fix
Replace the semaphore + O(n) try_lock scan with a per-pool free-list of worker slots carrying generation stamps — checked at dispatch, purged and reissued on respawn, so a crashed worker's stale slot can never route a request to a dead process and reject-not-queue is preserved through crash-respawn windows. the current permit-held-but-no-handle arm is already an error path; the redesign defines this behavior explicitly, with a property test over acquire/release/respawn interleavings
3 · AOT cache
The daemon precompiles each module once to ~/.riz/cache/cwasm/<blake3>-wt<wasmtime-major>-<config-hash>.cwasm; children Module::deserialize_file (one new unsafe with a SAFETY proof: host-generated file, hash-named, 0700 dir). Cache key includes wasmtime version + Config hash — .cwasm artifacts are version-specific — with recompile-on-mismatch fallback; tests cover truncation and version mismatch. Respawn drops from full Cranelift JIT to mmap.
4 · opt-in mux
worker_concurrency riz.toml key, default 1 everywhere — AWS Lambda guarantees one concurrent event per execution environment, and handlers legally rely on it (crud-accounts' module-level Map); that rationale ships verbatim as the semantic-departure warning. When opted in (bun/node only): N invocations on one event loop matched by frame id, per-worker bounded dispatch queues, crash accounting = one breaker increment per worker death (never per in-flight id), all in-flight ids resolve 502 — regression-tested. CANCEL applies to multiplexed JS workers; a sync wasip1 guest cannot observe it — kill is the only lever there, kill-when-idle preferred where the deadline allows. lands only with bench evidence — PR11
Benchmark discipline

The 91,419 req/s · p99 845µs headline is quoted on four public surfaces (benches/README, CONTRIBUTING, CAPABILITY-CARD, web/index.html) and benches/README.md:50 names stdin/stdout in its methodology. The bench re-run and every quoting surface move in one commit with the wire cutover, enforced via the claims registry — no stale-claims window. The CI-gated dispatch floor from #61 (tests/perf_http_floor.rs) must stay green through every transport PR — it is the structural-regression tripwire for exactly this kind of change.

Broker-path cost accounting: the UDS hop adds ~20–60µs versus the old in-child block_on — noise against millisecond backend I/O, repaid immediately by real connection pooling (N workers share max_connections instead of N cold clients) and per-function-correct limits.

Assurance

Testing strategy

  • Capability integration tests, one per capability, mirroring the proven wasm_broker_pg process-tree pattern (riz → __wasm-host → guest → UDS → daemon → mock backend): http (local origin asserting auth-header injection, origin pinning, denied absolute-URL attempt, too_large) · dynamo (mock that verifies SigV4 canonicalization before answering) · supabase (PostgREST-shaped mock) · clerk (local JWKS + minted JWT).
  • Broker cross-cutting: per-function limit tombstone (concurrency=4 workers, grant max_inflight=1, 4 simultaneous calls → exactly 1 admitted, 3 throttled) · scrubbed-env assertion · missing-secret-at-startup failure · closed-error-set exhaustiveness · daemon-kill mid-call desync (guest receives error{code:backend}, worker survives, next invocation succeeds) — merge-blocking, not follow-ups.
  • Transport: codec property tests (roundtrip, truncated header, oversize, unknown type) · wire-version fallback (RIZ_WIRE absent → line-JSON still answers) · multiplex correctness (distinct payloads on one worker, matched by id, no cross-talk) · crash-with-inflight (all ids 502, breaker +1 exactly) · backpressure ladder (saturation → 429, never queued) · AOT warm/cold/corrupt/version-mismatch · free-list property test over respawn interleavings.
  • Nextest discipline: the leak signal is load-bearing — the broker service gets an explicit shutdown path and every new child (scaffolded fleets, __wasm-host under the new topology) tears down deterministically; the two smoke binaries stay excluded from the default filter.

Risks

1 · aws_lambda_events may not compile to wasm32-wasip1

chrono/http feature graph. Mitigated: typed events are a feature, the Value path is guaranteed, CI verifies the target before the template ships.

2 · sqlx's tree may trip the deny.toml/SBOM gate

Mitigated: the PgBackend trait keeps the swap contained; deadpool-postgres fallback pre-agreed (open question 1).

3 · env_clear breaks workers relying on inherited vars

Mitigated: explicit per-runtime allowlist, escape hatch, all-six smoke + canary in the same PR.

4 · broker shares fate with HTTP serving

It moves into the gateway process. Mitigated by the supervised accept loop, bounded buffers, connection bounds, and the merge-blocking desync test.

5 · template_smoke_all builds three toolchains per CI run

Mitigated: cached target dirs, isolated binary outside the default filter, an explicit flake budget — otherwise it gets deleted in anger within a month.

6 · wire cutover invalidates published bench numbers

Mitigated by the atomic re-baseline rule and the #61 dispatch floor staying green.

7 · static conformance scanning is gameable and false-positive-prone

It is the fast layer only; the behavioral layer (template boot-smoke + parity matrix) is the real guarantee.
