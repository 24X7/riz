Provenance

This program follows the shape-purity work (PRs #66–#68 plus tranches 1–6). Its five items are the owner's compact directive of 2026-07-20, verbatim in intent: put real AWS examples in with a process to clone and test; rethink examples vs. scaffold templates and add more e2e examples; add an s3 broker for WASM; autogenerate OpenAPI docs like the MCP server and expose them as a route; add global environment variables. Decisions locked (owner, 2026-07-20)

AWS corpus lives as pinned live clones, not vendored snapshots — a manifest of (repo, commit, subdir); an isolated nextest binary clones each into scratch, builds, boots under riz, asserts, tears down, and skips cleanly when offline. Strongest claim ("their real code, pinned"), zero repo bloat. The two full-stack starters stay scaffoldable — riz new ai-chat keeps working; the templates/examples line is drawn with an explicit kind, not a directory-name guess.

Ground truth

The five asks

One directive, five separable deliverables. Each is a tranche; each ships independently green. C3 is the positioning centerpiece; C1, C4, C5 are self-contained; C2 is the keystone C3 hangs off of.

C1
Global environment variables. A top-level [env] table in riz.toml, merged into every function's environment, with [function.X.env] overriding it. Removes the per-function copy-paste that env scrubbing (tranche 6) surfaced.
C2
Templates ≠ starters ≠ examples. Separate fast-start scaffolds from read-only proof, and encode the distinction explicitly instead of by a starts_with("templates/") string check. Add more e2e examples.
C3
AWS conformance corpus + clone-and-test. Bring AWS's own official Lambda samples in as a pinned corpus and a harness that clones, builds, boots under riz, and asserts — proving riz runs the reference implementation's code unmodified.
C4
s3 broker verb for WASM. The broker has pg / http / dynamo; s3 is missing. Add get/put/list/delete_object, SigV4-signed host-side, WASM-only — the last capability the sandboxed guest can't self-connect for.
C5
OpenAPI document, autogenerated, as a route. Generate an OpenAPI 3.1 document from the same route table that already feeds the MCP tool list, and serve it at /openapi.json.
OUT OF SCOPE — stated, not omitted

Non-HTTP event sources. AWS's sample corpus is heavily SQS/SNS/S3-notification/DynamoDB-stream/ EventBridge-triggered. Those are out of riz's scope by decision. C3 pulls the HTTP / API Gateway v2 subset only, and the manifest records which event shape each sample targets.

Broker use by non-WASM runtimes. Bun/Node/Python/Rust/Go handlers are not network-sandboxed — they open their own S3/Postgres/HTTP connections. The broker (and the new s3 verb) exist for the WASM guest, the only network-sandboxed runtime. AWS ships no WASM samples, so nothing in C3 exercises the broker; C3 is a runtime-compat proof, C4 is a capability addition. They do not overlap.

A rendered docs viewer. C5 delivers the machine-readable /openapi.json only. A Swagger/ Redoc-style /docs HTML viewer is a natural follow-on but not in this program.

Current state

What four parallel recon passes established in source — the seam each tranche moves, and the one that is a genuine pre-existing bug.

Pre-existing bug, folded into C1

function_changed (hotreload.rs:448–459) does not compare the env map at all — so today, editing [function.X.env] and saving does not trigger a hot-swap; the change is silently ignored until a restart. C1 must extend this comparison, otherwise a global [env] edit would have the same dead-on-reload behavior. Fixing it here is correct scope: C1 is the env tranche.

C1 · self-contained · no dependency

Global environment variables

A top-level [env] table, folded into each function's environment in the config layer, so the child-process env assembly in pool.rs is left untouched and hot-reload re-merges for free. Precedence is decided entirely by stacking order — the layer applied last wins.

Because cmd.env(k,v) overwrites, layering global first, then per-function yields function-over-global precedence with no conditional logic. Broker vars (layer 3, set before) and riz internals (layer 6, set after) both remain unshadowable — the exact invariant the existing comments at pool.rs:209–211 protect. The scrub allowlist does not gate [env]: the allowlist only decides which daemon vars survive env_clear() (apply_base_env, layer 2). A global [env] is operator-declared literal values from riz.toml — the same kind of thing as [function.X.env] — applied straight through, never filtered.

Implementation — config-layer merge

Mirror the existing Config::effective_cors_for (config.rs:999–1006), which already layers per-function [function.X.cors] over global [cors] — the identical shape.

// config.rs — Config gains a sibling of `cors`
#[serde(default)]
pub env: std::collections::HashMap<String, String>,   // top-level [env]

// new helper, modeled on effective_cors_for (config.rs:999)
pub fn effective_env_for(&self, name: &str) -> HashMap<String, String> {
    let mut merged = self.env.clone();                // global first
    if let Some(f) = self.functions.get(name) {
        merged.extend(f.env.clone());                 // function overrides
    }
    merged
}

The pool then reads the merged map where it reads cfg.env today (pool.rs:218) — one call-site change threading the merged result through, no new pool.rs env loop. Hot-reload re-parses and re-merges with zero extra work.

The reload fix (folded in)

Extend function_changed (hotreload.rs:448) to compare the merged env map, so a change to either global [env] or [function.X.env] triggers a hot-swap. This also fixes the standing bug where per-function env edits are ignored until restart.

Config example
[env]                 # applies to every function
AWS_REGION = "us-east-1"
LOG_LEVEL  = "info"

[function.search.env] # overrides just for `search`
LOG_LEVEL  = "debug"
Tests
global-only reaches child; function overrides global; broker var un-shadowable by a colliding global; riz internal un-shadowable by both; edit-to-[env] triggers reload. unit + one e2e reload assertion
C2 · keystone · precedes C3

Templates ≠ starters ≠ examples

Today the tagline is right — "templates are starting points; examples are proof" (examples/README.md:3) — but the code separates them by a single string check, is_template_row() = subdir starts with templates/ (template_fetch.rs:74). That collapses under its own edge case: two full-stack starters (typescript-todo, ai-chat) live under examples/ yet are registered as riz new sources. They are both roles at once, and a directory-name guess cannot express that.

The fix: an explicit kind, three distinct tiers

Replace the prefix guess with a kind field on each BUILTINS row (template_fetch.rs:31–70). Scaffoldability becomes a declared property, not an inference from path.

  • No behavior lost. riz new ai-chat and riz new typescript-todo keep working — they're now kind=Starter instead of "an examples row that sneaks past the prefix check."
  • The prefix hack dies. is_template_row() (template_fetch.rs:74) and its two consumers — print_template_list() (main.rs:218,226) and the conformance test (lambda_shape_conformance.rs:180) — switch to reading kind. print_template_list renders three sections instead of two.
  • The one-to-one lock stays. template_set_maps_one_to_one_onto_runtime_kinds (lambda_shape_conformance.rs:134) still asserts exactly one kind=Template per RuntimeKind. Starters and examples are exempt from the one-per-runtime rule by construction.
  • Authoring contract preserved. The Lambda-shape walker (lambda_shape_conformance.rs:52) keeps traversing templates/* and examples/lambdas/* — starters and the external corpus are outside its authored-in-repo remit.
  • More e2e examples (the "more examples" ask). Add at least one capability-exercising example that the current set lacks — a dynamo-or-s3-backed WASM handler that drives the broker end to end — so the showcase tier demonstrates the guest capability path, not just HTTP shape.
Brittle couplings a reshuffle must respect

full_stack_builtin_points_at_examples (template_fetch.rs:493) asserts the starter subdir is examples/typescript-todo — starters stay under examples/, so this holds. examples/riz.all.toml and smoke-all.sh reference handlers by hardcoded relative path (e.g. examples/lambdas/chat/index.handler); C2 does not move the examples/lambdas/* handlers, so those paths are untouched. The only code change is BUILTINS gaining a field and three call-sites reading it.

C3 · positioning centerpiece · depends on C2, benefits from C1

AWS conformance corpus

The load-bearing claim riz can make: it runs AWS's own reference Lambda code, unmodified. phynd-search proved it once for a real Rust lambda; C3 turns that into a pinned, repeatable, multi-runtime conformance suite the site can cite because it's tied to a test.

Delivery — pinned live clones (locked)

A manifest of (repo, commit, subdir, runtime, event_shape, assert) rows. An isolated nextest binary (tests/aws_conformance.rs, gated like e2e_smoke_all) walks the manifest, and for each row runs the pipeline below in scratch. Nothing is copied into the tree; the corpus is referenced, so the claim is "AWS's actual code at commit abc123," not "our copy of it."

  • Hermetic-enough. Pinned commits make runs reproducible; the binary skips (does not fail) when the network is unreachable or a toolchain is missing — the same skip discipline template_smoke_all uses per leg.
  • HTTP / API Gateway v2 subset only. Each manifest row records event_shape (apigw-http-v2 · apigw-rest-v1). riz emits v2; v1-authored samples are flagged and either excluded or noted as a known event-shape delta, not a silent pass.
  • Zero handler edits. If a sample needs a source change to boot, that is a finding, recorded in the manifest with the reason — not patched away. The proof is worthless if we edit the samples.
  • C1 makes the configs clean. Samples that read AWS_REGION / table names get them from a single generated global [env], not per-row duplication.

Starter corpus — the most authoritative sources

Rust
awslabs/aws-lambda-rust-runtime → examples/http-*. Same lambda_http crate phynd-search uses; runs on riz's Runtime-API path unmodified. highest signal — start here
Go
aws/aws-lambda-go → the APIGW-v2 proxy example. Second compiled-binary Runtime-API path.

Start with the two compiled-binary samples (Rust, Go — the cleanest signal), prove green, then extend to the stdio-adapter runtimes. Java/.NET/Ruby samples exist but are outside riz's runtime set and are not pursued.

C4 · self-contained · no dependency

The s3 broker verb

The broker already dispatches pg, http, and dynamo to the sandboxed WASM guest with credentials that never cross the boundary. s3 is the missing member. It mirrors dynamo almost exactly — SigV4-signed host-side, WASM-only enforced for free — with one real difference: S3 is a REST API, not a single POST with X-Amz-Target.

What differs from dynamo (the only real design work)

  • REST verbs, not one POST. get_object=GET, put_object=PUT, delete_object=DELETE, list_objects=GET on the bucket with a query string. The bucket/key live in the URL path, not a JSON TableName field.
  • Payload hash header. S3 SigV4 signs x-amz-content-sha256 over the body; add it to the signed header set. Service name flips "dynamodb" → "s3" in v4::SigningParams (template at dynamo.rs:198–206).
  • Endpoint shape. https://{bucket}.s3.{region}.amazonaws.com vs dynamo's https://dynamodb.{region}.amazonaws.com (dynamo.rs:75).
  • key_prefix maps directly. The existing grant field (config.rs:476–480) scopes which S3 keys a grant may touch — reuse enforce_key_prefix (dynamo.rs:235) unchanged in intent.
  • Read-only guard. get_object/list_objects read-only; put_object/delete_object mutating — rejected before signing under a read-only grant.
Free wins recon confirmed

SigV4 deps are already in Cargo.toml:137–139 — no new crates (and not aws-sdk-s3, which exists at Cargo.toml:86 for deploy-only artifact download; the broker hand-signs). WASM-only enforcement is generic (config.rs:1293–1301 checks RuntimeKind::Wasm for any grant) — s3 grants inherit it with zero new code. The key_prefix grant field already exists. Test model: tests/broker_dynamo.rs stands up a mock that independently re-signs each request and byte-compares the signature — copy it to broker_s3.rs.

C5 · self-contained · no dependency

OpenAPI, autogenerated, as a route

riz already walks its live function table three times — for the MCP tool list, llms.txt, and the registry JSON — reading the same fields each pass. An OpenAPI 3.1 document is a fourth pass of the same shape. And because OpenAPI 3.1 aligns with JSON Schema 2020-12, the hand-built schema fragments riz already emits transfer with minimal massaging. No utoipa/schemars dependency is added.

  • Mapping is mechanical. Each route's METHOD /path becomes an OpenAPI operation; inferred path params → parameters (path); typed [function.X.mcp.query] → parameters (query); [function.X.mcp].body is already verbatim JSON Schema → requestBody.content.schema as-is; lambda_response_envelope_schema() (encoding.rs:48) → the default response.
  • Served public at /openapi.json. One .route("/openapi.json", get(handler)) line in build_app (server.rs:39–47), mirroring /health and the existing A2A agent-card route (server.rs:248) — a public, unauthenticated, generated-JSON precedent. It describes the public HTTP surface, so public is correct; easy to move behind the shared bearer gate if ever wanted.
  • Positioning payoff. Every function is now simultaneously an MCP tool and an OpenAPI operation, both autogenerated from one route table, both served as routes — no annotations, no separate spec file to drift.
Delivery

Five tranches, sequenced by dependency

Each tranche is one PR (or a tight pair), independently green, merged the usual way. Only two edges are real: C2 must precede C3 (the corpus lands in the tier C2 defines), and C1 should precede C3 (clean sample configs). C4 and C5 float free.

Gates — every tranche, before push

cargo fmt --all -- --check · cargo clippy --workspace --all-targets -- -D warnings · ./scripts/safety-check.sh --gate · the fast nextest set · then the isolated binaries (e2e_smoke_all, template_smoke_all, perf_regression, chaos, and — new in T3 — aws_conformance). cargo-nextest only. T2 and T4/T5 touch the runtime-count source-of-truth: any change to the six-runtime enumeration updates every prose site in lockstep (rg -n "runtimes").

Assurance

What proves it, what could sink it

Proof per tranche

  • T1 — unit tests for the four precedence cases + one e2e that edits [env] and asserts the running child sees the new value (closes the reload bug).
  • T2 — the existing one-to-one and Lambda-shape conformance tests stay green under the new kind; a test asserts riz new ai-chat still resolves; print_template_list renders three tiers.
  • T3 — aws_conformance boots each pinned AWS sample and asserts a real response; runs green in CI where toolchains exist, skips (not fails) offline. This is the citable "runs AWS's own code" evidence.
  • T4 — the re-signing mock proves byte-exact SigV4; read-only and key_prefix rejection tests prove the guardrails fire before signing; a WASM example (from T2) drives it end to end.
  • T5 — a test fetches /openapi.json and validates it parses as OpenAPI 3.1 with one operation per declared route.

Risks

T3 network
AWS repos move or a pinned commit disappears. Mitigation: pin commits (not branches); skip-on-unreachable; a clear failure message naming the dead ref. Vendoring stays the documented fallback if flakiness proves chronic.
T3 v1 vs v2
A sample authored for REST API (v1) hands the handler a different event than riz's v2. Mitigation: event_shape in the manifest; v1 samples excluded or logged as a known delta — never a silent pass.
T4 S3 quirks
S3's signing has more surface than dynamo (payload hash, path-style vs virtual-host, query signing for list). Mitigation: the independent re-signing mock is the oracle — if our sign and the mock's re-sign disagree byte-for-byte, the test fails.
T2 drift
Renaming/moving a starter breaks riz new and a test asserting its subdir. Mitigation: C2 keeps starters under examples/ and only adds a field — no relocation.

Scope guard restated

This program adds runtime-compat proof (C3), one capability (C4), and three DX/surface refinements (C1, C2, C5). It does not expand riz beyond HTTP/WS Lambdas, does not route non-WASM traffic through the broker, and does not ship a docs viewer. Everything here is verified against main @ 7f72bc3; nothing is built until this spec is approved.
