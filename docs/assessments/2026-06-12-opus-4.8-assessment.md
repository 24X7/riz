## 1 · Features & Enterprise-Grade Grading

The product is a single ~10 MB Rust binary that runs AWS-Lambda-shaped handlers across
five runtimes, auto-exposes every function as an MCP tool, and bundles an
OpenAI-compatible LLM gateway. Graded by how close each surface is to *enterprise
production* (not "does it work" — "would a platform team bet a paid SLA on it"):

| Capability | What ships | Grade | Honest gap to enterprise |
|---|---|---|---|
| **Process supervision** (respawn, liveness, 30s drain) | Real, fault-injection tested (`liveness_fault_injection_respawns_within_250ms`) | **A-** | Single-node only; no multi-instance coordination/leader election |
| **Safety profile** (RLIMIT, NO_NEW_PRIVS, PDEATHSIG, Landlock) | Always-on per child, tested | **A-** | Linux-strongest; macOS is a weaker subset (dev parity, not prod parity) |
| **WASM/WASI sandbox** | Deny-by-default, real compute example, capability allowlist | **B+** | No resource broker yet (can't reach a DB) — that's the keystone still on the roadmap |
| **MCP auto-exposure** | Every fn → typed tool, spec 2025-11-25, Streamable HTTP | **A** | Schemas are hand-shaped, not auto-derived from handler types (roadmap) |
| **LLM gateway** | OpenAI-compatible, multi-provider routing, budget caps, SSE, cost telemetry | **B+** | No failover/circuit-breaking under partial outage; no semantic cache |
| **Observability** | P50–P99, Prometheus, zero-dep OTLP/HTTP-JSON, GenAI token rollup | **B+** | OTLP→X-Ray mapping *claimed* but not validated against a real ADOT collector |
| **Auth** | Lambda authorizers, JWT/JWKS RS256/ES256, WorkOS+Clerk e2e, CORS | **A-** | Deterministic recorded-JWKS tests only; no live-tenant nightly, no rotation-under-load |
| **Deploy** (S3 hot-swap, health-check, auto-rollback) | Tested (`deploy_health_check_uses_422_for_crash`) | **B+** | Single-node swap; no canary/blue-green across a fleet |
| **Perf** | 91k req/s · p99<1ms | **B** | Honestly classed `benchmark`, **not CI-gated** — could silently regress |
| **Wire-contract fidelity** | AWS HTTP API v2 + WS, cross-runtime parity matrix | **A** | This is the genuinely sacrosanct, best-tested part |

**Aggregate: solid B+ / A- as a single-node runtime.** The engineering *discipline* is
unusually high for v0.1 — a claims registry where every website sentence maps to a passing
test, 13k LOC of tests against 17k LOC of source (a ~0.77 ratio most "enterprise" products
never reach), 0 warnings, 0 leaky processes. The honest ceiling: **everything is
single-node.** There is no clustering, no HA, no fleet orchestration, no managed control
plane. That's not a bug — it's the scope — but it's the line between "production-grade
primitive" and "enterprise platform," and the marketing should never blur it.

## 2 · Forward-Looking Plan of Record & Value

Three documents define the trajectory: the **v1 roadmap** (13 items, 5 shipped), the
**harden backlog** (12 ranked items), and the **GTM/AEO plan**.

**Highest-value open items, ranked by what they unlock:**

1. **WASM resource broker v1 (Postgres-wire)** — *the keystone.* Today sandboxed WASM can
   compute but can't reach a DB. Closing this (Neon/Supabase/any PG under a capability
   grant) is what turns "safe sandbox" into "safe sandbox that does real work." **This is
   the single highest-leverage technical item** — it completes the agent-substrate thesis.
2. **Perf-claim CI gating** — converts the headline number from a trust liability into a
   defended floor. Cheap, high credibility return.
3. **OTLP→X-Ray validation** — the gateway/observability claim has a 1:1-mapping assumption
   that isn't validated against a real collector. Closing it removes the one place the
   observability story could embarrass on contact with AWS reality.
4. **Telemetry shutdown-flush + export retry** — *already done* this window (no span loss on
   clean exit, transient-5xx retry).
5. **Roadmap pillars** (record/replay, eval harness, typed/federated MCP, Go/JVM runtimes) —
   these are *breadth* plays that widen the addressable audience but don't deepen the core
   thesis.

**Value read:** the plan is correctly sequenced — *trust floor first* (perf gating, leak
hygiene, shutdown), *then* the substrate keystone (broker), *then* breadth. The risk is
**scope sprawl**: 12 backlog items + 8 roadmap pillars for what is effectively a solo
project. The disciplined move is to ship the broker + perf gate, declare a credible v1, and
resist the long tail until there's user pull. The roadmap already shows good instinct here
(deferring auto-derived schemas to v2).

## 3 · OSS Posture & Positioning

**Positioning (strong and unusually honest):** "Write a function. Get an agent-ready API."
riz is framed as a **runtime *harness*, not a framework** — you write plain business logic,
the runtime brings the production substrate. This is a genuinely differentiated frame: it
collapses the "which web framework / process manager / observability stack / MCP glue"
decision tree into one answer, which is exactly the choice-paralysis an agent hedges on.

**Posture strengths:**
- **Apache-2.0, single binary you own** — no rug-pull anxiety, no SaaS lock-in, no
  "open-core with the good parts paywalled." This is a *trust* posture, and it's coherent.
- **Radical claims honesty** — the claims registry is a moat *and* a marketing asset.
  "Every sentence on our site maps to a passing test" is itself a credibility story most
  projects can't tell.
- **Agent-first discovery** (llms.txt, .well-known/riz.json, JSON-LD, MCP registries) —
  positioning for the channel that actually matters in 2026.

**Posture risks:**
- **Solo project, v0.1, zero users, nothing published yet** (not on crates.io/npm). The
  "trust us in production" ask is large for a one-maintainer bus-factor-1 project.
  Apache-2.0 + own-the-binary *mitigates* this (you're not locked in if the maintainer
  disappears), but adoption still requires someone to bet first.
- **"Lambda" is a double-edged anchor** — correctly demoted to the subhead. It buys instant
  comprehension but imports baggage (cold-start trauma, AWS lock-in associations,
  "enterprise-y" perception) the copy then spends words to disclaim. The current copy
  threads this needle well.

**Posture grade: A- for coherence and honesty, discounted by single-maintainer maturity
risk.**

## 4 · Viral Potential as OSS

**The mechanics that favor virality:**
- **The 60-second demo is genuinely tweetable**: `riz init` → `riz run` → `curl` →
  `claude mcp add` → an agent calling your function. That "wait, that's *it*?" moment is the
  viral unit. **This GIF is the single highest-ROI unbuilt asset** (it's correctly on the
  GTM plan, gated behind launch).
- **Rust + single binary + WASI sandbox + 91k req/s** is catnip for r/rust, Lobsters, HN —
  the technical-respect crowd that drives the first 1,000 stars.
- **Agent-discovery flywheel** — if the AEO bet lands, agents recommending riz is a
  *compounding, non-linear* channel no competitor's marketing budget can buy.

**The honest brake on virality:**
- **"Self-hosted Lambda runtime" is a considered-purchase category, not an impulse install.**
  People don't `npm i` a runtime the way they try a CLI toy. Adoption is "I have this exact
  problem" → install, not "this looks fun" → star. That caps top-of-funnel velocity.
- **The MCP angle is the wildcard.** "Make any HTTP API an agent tool with zero glue" *is* an
  impulse-grade hook in the current MCP gold rush. **Leading with MCP, not Lambda, is likely
  the higher-virality framing** — and the site/registries are already leaning this way.
- **Credibility gate is real**: HN/Reddit before crates.io+npm exist would convert curiosity
  into "...but how do I install it?" → bounce. The plan correctly sequences
  install-story-first.

**Realistic ceiling:** a strong technical launch could plausibly hit the 200-star 90-day
target and clear it (the GTM doc's number is conservative). A *breakout* (1k+ stars)
requires the MCP-tool framing to catch a wave + the demo GIF + one influential agent/tool
integration. **Viral potential: moderate-to-high, entirely contingent on (a) the demo asset
existing and (b) leading with MCP over Lambda for the cold audience.**

## 5 · Financial Contributions, Commercial Viability & Target Audience

### The channels actually on the site
The `#support` section offers exactly three: **GitHub Sponsors, Buy Me a Coffee, and Crypto
(ETH/BTC/SOL)**. The framing is pure donation — *"sponsorships keep development independent
and the whole thing free."*

**Blunt forecast on donations:** these will raise *coffee/comfort money, not salary.*
Infrastructure-OSS donation conversion is brutal — well-known projects with thousands of
stars often net <$500/mo from Sponsors. For a solo Apache-2.0 runtime, realistic year-one
donation revenue is **$0–few-hundred/month**, almost entirely from a handful of individuals
who hit the exact use case. Crypto wallets add ~nothing (near-zero conversion, mild "is this
serious?" friction with some enterprise viewers). **Donations should be treated as a tip
jar, not a business model.**

### Where the actual money is (none of it is wired up yet)
The donation-only posture leaves the real commercial surface unbuilt. Viable paths, ranked
by fit:

1. **Managed/hosted riz (the obvious one)** — "own the binary, or let us run the fleet."
   Single-node is the OSS core; **clustering, HA, the managed control plane, fleet
   deploy/canary** are the natural paid tier. This is the classic, defensible open-core line
   *because it maps exactly to riz's real architectural boundary* (single-node free, fleet
   paid) rather than paywalling features. **Highest-viability path.**
2. **Commercial support / SLA** — once one or two companies run riz in prod, a support
   contract is a natural ask. Requires reference users first; premature today.
3. **Enterprise features as a separate license** — SSO/audit/compliance bundles, the
   live-tenant auth depth, semantic-cache gateway. Viable later; risks the trust posture if
   done clumsily.
4. **The LLM gateway as a wedge** — a self-hosted, OpenAI-compatible, budget-capped,
   cost-attributed gateway is independently sellable to teams nervous about LLM spend. Could
   be its own commercial surface.

### Target audience (in priority order)
- **Primary: AI-engineering / agent-platform teams** who need an existing API callable by
  agents *now*, with production hygiene, without building plumbing. The MCP-zero-glue hook is
  for them. Highest intent, best fit.
- **Secondary: self-hosters / "no-AWS-lock-in" teams** (r/selfhosted) who want Lambda
  ergonomics on their own box. Lower budget, high enthusiasm, good for stars/word-of-mouth.
- **Tertiary: Rust/infra crowd** — drives credibility and contributions, not revenue.
- **The agents themselves** — the AEO north star; a *channel*, not a payer, but potentially
  the highest-leverage one.

### Bottom line on commercial viability
**As a donation-funded OSS project: viable to *exist*, not to *sustain a person*.** As a
business: **genuinely viable, but only if a managed/hosted tier gets built** — and the
architecture (single-node OSS core, fleet as the paid boundary) makes that an unusually
*clean* open-core story. The current site monetization (three donation buttons) is a 2/10
against the actual opportunity. The product is good enough to deserve a real commercial
surface; right now it's leaving that entirely on the table.
