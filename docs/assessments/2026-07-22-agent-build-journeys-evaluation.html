Agent build-journey evaluation

Nine agents landed on riz.dev cold and actually built — following the site's own instructions, on the real riz 0.3.0 binary. Four language tracks (TypeScript · Python · Go · WebAssembly-Rust), each in two scenarios: adapt an existing AWS Lambda unchanged, and build brand-new from scratch. Every scaffold, build, boot, and call was executed and its output recorded. This is the tailwinds, the headwinds, the must-do fixes, and the roadmap that came back.

Headline

The core promise held on every track. An unmodified AWS-Lambda-shaped handler — TypeScript, Python, Go, and Rust — ran on riz with only a riz.toml and no code change, and an agent landing cold on just llms.txt + /.well-known/riz.json could install, create, run, and call a function end-to-end. The wedge works. What's rough is the getting-started surface for humans, a real Python bare-handler crash, and a cluster of CLI ergonomics — none of which touch the runtime contract.

Verdict

The foundation is solid; the on-ramp is where we lose people

Read the two columns of the matrix below as the product's two wedges. The adapt-an-existing-AWS-Lambda column is riz's sharpest claim — "your handler runs unmodified" — and it is literally true in TS, Python, Go, and Rust (a byte-for-byte official AWS Rust sample returned the exact expected body). The brand-new column is a one-command scaffold that works first try for every runtime. The friction is almost entirely in the words around the commands: the human quickstart leads with a TUI a headless caller can't run, two documented handler forms don't match what ships, and the flagship "unmodified AWS binary" path — the strongest claim — is the only one with no walkthrough.

The journeys

Eight builds, all executed for real

Each cell was run on the 0.3.0 binary: scaffold or adapt → build → boot headless → curl → confirm the response. "Adapt existing" wrote a canonical, unmodified AWS handler and wired it in with only a riz.toml.

Tailwinds

What's working — lean into these

Verified live, not asserted. These are the parts of the experience that already convert a skeptic.

  • The unmodified-AWS-handler promise is literally true — all four tracks. A canonical TS/Python/Go handler and a byte-for-byte official AWS Rust binary each ran with only a riz.toml; the AWS Rust sample returned the exact expected body, status, and content-type — no riz import, no shim. rust-existing: awslabs/aws-lambda-rust-runtime http-basic (pinned) → lambda_http 1.3.0 → ran on riz
  • The machine surface is genuinely self-serve for an agent. llms.txt and riz.json both lead their 3-line adopt block with headless riz run; MCP tools/list returns typed input/output schemas and self-corrects with -32602. An agent can install → create → run → call cold, no HTML parsing. discovery: initialize (2025-11-25) → tools/list → tools/call {name} → "hello, mcp-agent" from the real handler
  • riz doctor is a standout confidence-builder. Validates the toml, confirms the runtime binary on PATH (abs path), resolves the exact handler file, and probes the port — green before boot, every track.
  • Scaffolds are fast and self-consistent. riz new <t> fetches from git in <1s, exit 0, and README + riz.toml + Cargo.toml + src all agree; the riz-wasm authoring model is exactly as advertised — fn handler(event, ctx) + riz_wasm::run, no stdin plumbing, no unsafe.
  • No false advertising. /openapi.json serves OpenAPI 3.1 with the live route; opt-in surfaces (/.well-known/agent-card.json, /llms.txt) correctly 404 until their [agent]/[static] blocks are configured — capabilities appear only when enabled.
  • Pre-flight + probes are clean. riz routes and riz validate reinforce doctor; /health and /ready are open and instant — a clean liveness signal for an orchestrator.
Headwinds

Where it fought us — the recurring friction

Ranked by how many independent journeys hit it. The first one is systemic: it showed up in every single journey.

  • The human quickstart boots with a TUI a headless caller can't run. (8/8 journeys) start.html's "copy all" block and step 3 both make riz --dev THE run command. A headless agent / CI / no-TTY shell copying it launches a Ratatui TUI that fails; the working riz run lives only in prose further down. The machine surface gets this right — the primary human page does not. start.html:70 (copy-all) + start.html:74 use riz --dev; riz run only at start.html:165
  • Python's documented handler form crashes with an opaque 502. llms.txt and the docs field-table say handler = "app.lambda_handler" (the AWS-canonical bare form). Booting that returns 502 "process error: Broken pipe"; the child's real stderr (No module named app) is swallowed. Only ./app.lambda_handler works — a form that appears nowhere on the site. py-existing: root cause src/process/python.rs:63-68 (.with_extension("").canonicalize() on a path that can't exist)
  • Compiled-runtime scaffolds tell you to boot before you build. riz new go's printed "Next steps" is cd … / riz run — no go build. Booting hits a ./hello binary that doesn't exist yet. The template README has it right; the CLI's own output contradicts it. go-new: src/main.rs:415-434 prints a build line only when Cargo.toml is present — Go (go.mod) gets none
  • Doc ↔ reality mismatches on handler shape. Node's index.handler → file index.mjs (not .js) is stated only in llms.txt; the Python template ships main.py while llms.txt says app.py. A newcomer bringing their own file name gets a file-not-found until doctor/boot tells them.
  • CLI port ergonomics fight muscle memory. riz run -p 3987 is rejected — the port is a top-level flag (riz -p 3987 run), documented for --dev but never for -p. And riz mcp inspect hardcodes :3000, ignoring the project's [server] port, so the documented verify step fails on any other port. discovery + wasm-new: `riz run -p` → "unexpected argument -p"; src/main.rs:56 port is a Cli field, not a run arg
  • The flagship claim has no walkthrough. "Run an unmodified official AWS Rust/Go binary" — riz's strongest differentiator — is only asserted (a ledger sentence + a CI test name + an llms bullet). The novel WASM path is a one-command scaffold, so paradoxically the headline compatibility story is the least documented, task-wise.
  • First boot looks broken to a log-tailer. Every fresh scaffold logs a top-line ERROR: SECURITY: /deploy has no auth configured — the safe default for an endpoint the template never uses, at the wrong severity. src/main.rs:1343 — tracing::error! on a benign unconfigured-deploy default

Roadmap

Bigger bets, once the fixes land

The fixes above make the current on-ramp correct. These make it excellent, especially for the two audiences riz is actually chasing: agents and Lambda refugees.

  • Machine-readable everything. --json on new --list, routes, doctor, validate so an autonomous agent never has to scrape a human-formatted table. The MCP surface already sets this bar; the CLI should match.
  • A first-class "bring your own AWS Lambda" adopt path. Getting-started is scaffold-first, yet the sharpest wedge is "your existing handler runs unmodified." A guided riz adopt ./my-handler (or a docs track) that generates the riz.toml for an existing handler would sell the wedge directly.
  • Handler-resolution parity with AWS. Accept bare module names (app.lambda_handler), put the project root on the module path — so a Lambda dev's muscle memory transfers and the ./ vs bare distinction disappears.
  • First-boot cleanliness as a guarantee. A fresh scaffold should boot with zero ERROR/WARN noise at INFO. Treat "hello-world boots clean" as a tested invariant.
  • Surface child crashes. Pipe the handler process's stderr into the server log and the --dev inspector, so a crash reads as "No module named app", never "Broken pipe".
  • Consider riz build / build-aware riz new. For compiled runtimes, run (or offer to run) the build as part of scaffolding, so "scaffold → run" is true for every runtime, not just the scripted ones.
  • Port ergonomics that match every other CLI. riz run -p 3000 should be the obvious, working form; arg-order rigidity is a papercut an agent hits on its very first collision-avoidance move.
Method

How this was run — so we can re-run it

A deterministic multi-agent workflow. One discovery agent (agent's-eye view of the machine surface) plus eight build-journey agents (four tracks × two scenarios), each in an isolated scratch dir on the real riz 0.3.0 binary, following the local site copy (web/*.html, llms.txt, riz.json) as "riz.dev." Every command was executed and its output recorded; each agent cleaned up its build artifacts. Findings were returned as structured JSON, and every load-bearing bug claim in the fix backlog was re-verified against source before landing here.

  • Scope: HTTP / API Gateway v2 handlers, all six runtimes' concerns via the four language tracks. WebSocket, the LLM gateway, A2A, and guards were exercised by the discovery pass but are not the subject of these journeys.
  • What "adapt existing" meant: a canonical, unmodified handler of the kind a team already runs on AWS (Node exports.handler, Python lambda_handler, Go lambda.Start, Rust lambda_http), wired in with only a riz.toml. For Rust, AWS's own pinned sample.
  • Re-run target: when the P0–P1 fixes land, re-run the same nine journeys and expect the Python cell to go green, the ERROR log to vanish, and the -p / next-steps snags to clear — a measurable before/after on this exact matrix.
