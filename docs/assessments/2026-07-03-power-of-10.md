# Power of 10 — safety review loop

Recurring semantic pass over what the lints cannot see (docs/SAFETY.md,
enforcement tier "review"), plus the ratchet burn-down. One iteration per
module group; counts from `scripts/safety-check.sh` (production code,
`--lib --bins`, unique violation sites).

### Review-tier findings (rules the lints cannot see)

| Area | Risk profile | Rule | Finding | Disposition |
|---|---|---|---|---|
| `a2a.rs` task store | request path | 3 | **Unbounded growth from remote input.** Every `SendMessage` inserted a `TaskEntry` into a `DashMap` that was never pruned — a remote client could grow the store without limit; entries hold snapshots, broadcast channels, and join handles | **Fixed**: `TaskStore` with `MAX_RETAINED_TASKS = 1024` — at the cap the oldest *terminal* task is evicted; if all retained tasks are live the new task is rejected with a JSON-RPC error (running state is never dropped). Unit-tested both branches |
| `ws::Connection.outbound` | request path | 3 | The per-connection outbound queue was unbounded *by documented design* ("v0.1 NOTE … revisit"); a slow WS client plus a chatty handler = OOM. The MCP ephemeral session inherited this | **Fixed** (cross-module, minimum edit): bounded `mpsc::channel(OUTBOUND_CAPACITY = 256)`. Backpressure is explicit: `@connections` POST answers **429** on a full queue (AWS parity — PostToConnection's `LimitExceededException`), `Close` frames use best-effort `try_send` on teardown paths. Unit-tested (POST → 429 at capacity) |
| `a2a.rs` `send_message` wait loop, `stream_frames` loop | request path | 2 | Bare `loop {` sites | Pass: both await every iteration and exit on a deadline (`sleep_until` / `timeout_at`), terminal state, or channel close. Deadline arithmetic now saturating |

| `a2a.rs` agent-loop `messages` | request path | 3 | Transcript grows per hop: bounded by `max_hops` iterations × per-response tool calls; item sizes bounded by the gateway/provider response and function response sizes (both HTTP-bounded) | Accepted: bounded in the aggregate by config; an explicit byte cap would add complexity without changing the bound's existence. | A2A ↔ workload cycle | request path | 1 | A user function *could* HTTP-call its own `/_riz/a2a`, creating a delegation cycle that the `riz-a2a-hop` header does not track (the header only propagates through `delegate_to_*` tools) | Accepted with rationale: each leg is a spawned task (no stack recursion — rule 1's letter is satisfied), and each consumes pool-semaphore slots and `task_timeout_ms` budgets, so the cycle degrades into task-store/backpressure rejections rather than resource exhaustion. ### Behavior changes (all covered by new tests)

1. Empty `choices` from the LLM gateway: task fails with a status message
   instead of panicking the task's tokio task (`a2a.rs`).
2. `POST /_riz/connections/{id}` on a backlogged connection: **429** instead
   of unbounded buffering (`ws/management.rs`;
   `post_returns_429_when_outbound_queue_full`).
3. A2A task store at capacity: oldest terminal task evicted, or the new task
   rejected with −32603 when all 1024 retained tasks are live (`a2a.rs`;
   `task_store_rejects_insert_when_full_of_live_tasks`,
   `task_store_evicts_oldest_terminal_task_at_cap`).
4. Pathological timeout configs clamp instead of panicking
   (`deadline_after_saturates_instead_of_panicking`).

Everything else is behavior-preserving: the `serde_json` `[]` → `.get()`
conversions are observationally identical (`Value`'s `Index` returns Null for
missing keys), and the function splits move code without changing it.

### Behavior changes (all covered by new tests)

1. An operator-supplied `Cache-Control` value that is not legal HTTP: empty
   **500** instead of a request-task panic, and the bad value cannot inject
   a header (`malformed_cache_control_config_returns_500_not_panic`).
2. `Last-Modified` is formatted by the `httpdate` crate; a `SystemTime`-
   overflowing mtime omits the header rather than panicking
   (`last_modified_is_valid_imf_fixdate` round-trips the format).
3. Everything else is behavior-preserving, now pinned where it was not:
   Range values at and beyond `u64::MAX` answer 416
   (`extreme_range_values_return_416_not_panic`), oversized suffix ranges
   serve the whole file (`oversized_suffix_range_serves_whole_file`), and
   two proptests fuzz `parse_single_range` with arbitrary header bytes and
   file sizes, asserting no panic and in-bounds, internally consistent
   results (rule 10's "all analyzers" in spirit).
